# Rules for working with LLMs on the Go client (`client/go/`)

## Pre-push checks

The repo does **not** have a pre-commit hook for Go code in `client/go/`. CI
runs `golangci-lint` (`v2`) which enforces `gofmt`, `staticcheck`, and the
other linters configured in `client/go/.golangci.toml`. Run these locally
before pushing:

```bash
cd client/go
bazel run @rules_go//go -- fmt ./...   # invokes gofmt via the project's Go toolchain
bazel run @rules_go//go -- build ./...
bazel run @rules_go//go -- test ./... ./test/...
golangci-lint run                      # if installed; CI runs this regardless
```

The `bazel run @rules_go//go --` form invokes the Go toolchain that the
build uses, so it works without a system-wide Go install. If you already
have `go` and `gofmt` on your PATH, you can use those directly instead.

Common pitfalls that CI will catch:

- **Trailing line comments on consecutive lines must be column-aligned.**
  When two adjacent lines (no blank line between) both have trailing
  comments, `gofmt` pads the shorter one with spaces so the comments start
  at the same column. The reverse also applies: don't manually align
  comments across non-consecutive lines or across blank lines. Just run
  `bazel run @rules_go//go -- fmt ./...` and let gofmt handle it.
- **`staticcheck` enforces `SA1019`** — calling a `// Deprecated:` symbol
  triggers an error. For SDK-internal call sites that legitimately need a
  deprecated API, prefer rewriting to use the unexported backing field. For
  test code that intentionally exercises the deprecated surface, suppress with
  `//nolint:staticcheck // <reason>` at the end of the offending line.

## Deprecating a public symbol

1. Add a `// Deprecated: <replacement>` paragraph as the last block of the
   doc comment on the symbol.
2. Update every internal caller (in `client/go/`) to stop using the symbol —
   typically by reading the unexported backing field directly, since
   `staticcheck` will otherwise flag the internal use.
3. Update tests that exercise the symbol with `//nolint:staticcheck` and a
   short reason.
4. Add a note to `client/CHANGELOG_GO_JS.md`.

## Public API conventions

`client/go/public_api_test.go` enforces that every public method on an
exported type ends in a `*XxxParams` argument. Any new method that does
not follow the convention must either:

- Be added to `skipMethods` with a one-line reason explaining why (e.g.
  accessor, lifecycle signal, interface implementation), or
- Replaced with a `*XxxParams`-shaped signature.

`knownParamsViolations` is a frozen set of pre-existing exceptions; do not
add to it.

## Internal-only helpers

If a function needs to be reachable from multiple files in `package modal`
but is not part of the public surface, leave it unexported (lower-case
name). Same-package access is fine across files.

## Protobuf

`client/go/proto/modal_proto/` is generated by `client/go/scripts/gen-proto.sh`
and **is** checked in (unlike the internal `go/` SDK). Regenerate when the
underlying `.proto` schema changes, and commit the resulting `*.pb.go`.

## Bazel BUILD files

Adding, renaming, or deleting a `*.go` file in this directory requires
regenerating BUILD files. From the repo root:

```bash
bazel mod tidy
bazel run //:gazelle
bazel mod tidy
```

Commit any resulting `BUILD.bazel` changes alongside the source change. CI
runs a Gazelle check that fails the build if BUILD files are out of sync.
See [`bazel-keep-in-sync`](../../.claude/skills/bazel-keep-in-sync/SKILL.md)
for the full procedure.
